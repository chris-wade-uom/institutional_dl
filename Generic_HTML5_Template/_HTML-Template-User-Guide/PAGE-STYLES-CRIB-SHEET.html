About

This template brings together an HTML5 structure with CSS styles with the aim of supporting the development of on-line teaching notes. It aims to keep page structure simple, but provide a strong underlying framework which will take care of providing clean presentation, many of the accessibility requirements and responsive behaviour to allow content to display equally well on mobile, tablet and desktop screens. It does not provide all the complex styles of a fully formed website e.g. column layouts. Key features include:

  • Clear styling of HTML5 standard elements e.g. header tags, lists, tables.
  • Optimisations to ensure content displays cleanly on small and large screen devices, e.g. images, tables and videos.
  • Optimisations to improve the user experience when content is hosted in a CMS / VLE, (see notes below).
  • Optimisations for accessibility and advice on good practice throughout these notes. e.g. colour contrast, links.
  • Libraries (pre-configured) to support the inclusion of Mathematical notation and Flash.
  • A small set of clean, logically designed custom styles to allow you to add enhancements to document presentation, with specific consideration given to the context of Higher Education, e.g. callout boxes, list numbering formats and styling, display of code, and tabular data. Where appropriate, licences for icon packs have already been obtained.
  • An underlying HTML5 structure which uses semantically correct tags to describe the content (important for accessibility), e.g. use of <header>, <nav>, <main> and <footer> blocks.

Mobile platform support

Whilst your content may currently not be deployed on platform which is viewed on mobile devices, that might change in the future. This template attempts to take care of most of the hard work in supporting smaller screen devices.

  • Images and videos will automatically scale down to fit narrower screens.
  • Tables with many columns will scroll horizontally and where this occurs it is highlighted to the user to ensure data isn't accidentally missed.
  • Presentational "callout boxes" are designed to ensure they adjust to narrower screens as necessary.
  • Blocks of Mathematical notation will line-wrap should it not fit a narrow screen.

Note that whilst advice on the inclusion of Flash based media is provided, it is noted this is not supported on mobile platforms and appropriate advice is given. Appropriate recommendations are given for other media types to help ensure good cross-platform support of content.

Optimisations for CMS/VLE platforms

There are two notable optimisations to improve the experience of users when documents produced from this template are presented inside a CMS/VLE.

  • This template includes a "print view" option to "break out" of any CMS/VLE content frame and present the document content with styles better suited to print media e.g. font sizing, removal of some background colours. This allows the page to then be printed without being adversely affected by framework of the CMS/VLE.
  • Where javascript is enabled, as the template loads, it will check to see if it is hosted inside a secure HTTPS host and will fix and problematic insecure HTTP links within the content which would otherwise fail to open due to normal browser security restrictions.

Accessibility

It is important to ensure content is accessible to all users, including those who rely on assistive technologies such as screen readers. However, it need not be a burden. It is often the case that small changes you can make to improve the accessibility of your content will benefit all users, not just those with additional requirements. This template aims to deal with some of these responsibilities already and this guide provide good common sense advice for everything else. Throughout this guide there are "Best Practice" notes which include concise notes on what to think about to improve the accessibility of your content. We've gone to the trouble of wading through other best practise information so you don't have to.

Mathematics

Within many disciplines, the display of mathematical notation in teaching materials is a fundamental requirement. This template incorporates the MathJax library and has been pre-configured to suit a more common requirements of the HE environments. The configuration file is abstracted to a single external javascript file to allow changes to be made to an entire set of pages from a single place. Note that the mhchem extension for typesetting chemical molecular formulae and equations is not included by default, but can be easily enabled.

Technical note about this template guide

This template focuses on content tags that are most likely to be of use in the development of Teaching and Learning content. HTML5 tags which either: (1) encourage bad accessibility practice; (2) aren't relevant because they are designed to act as javascript hooks for richer web applications; or (3) are intended for building data forms for submission to a server, are all omitted from discussion within this guide as they are deemed not relevant.

Best practice

Always use correct headings, lists and other structural elements to provide meaning and structure to web pages. This helps with making your document more accessible to assistive technologies and also facilitates keyboard navigation within the page.

However being disciplined over using semantic correct HTML tags will provide a highly consistent presentation style throughout your document and so all users benefit. From a technical standpoint it also provides a central point to change those styles via the associated style rules document (CSS).

An excellent guide to explaining best practice for structuring text based web content with semantically correct elements is available at http://webaim.org/techniques/semanticstructure/.

Best practice

Lists are great from an accessibility standpoint because they provide structured order to content in a linear fashion and therefore add the all important semantic meaning to the enclosed content. Lists are recommended as potential replacements for simple tables, as tables can be more difficult to navigate and sometimes we provide info in tables that really would be better suited to lists. You should never rely on indentation to provide a visual list, use a proper HTML structure instead e.g. either an unordered <ul>list; an ordered (numerical) <ol> list; or a definition <dl> list.

Tables

HTML tables should be used to present tabular information in a grid, or matrix, with columns and rows that show the meaning of the information in the grid.

Warning - do not use tables to control page layout

Prior to the creation of CSS, HTML <table> elements were often used as a method for page layout. This usage has been strongly discouraged since HTML 4, and the <table> element should not be used for layout purposes. You can read more about why use of tables in webpage layout is so discouraged at http://webaim.org/techniques/tables/.

A basic data table layout should consist of the parent <table> tab immediately followed by a <caption> tag to describe the content of that table.

<table>
	<caption>Table description goes here</caption>
    ...

Following the caption, the table should include the different blocks that make up its structure - a table header block <thead>, if required an optional table footer block <tfoot> would follow, and finally the main table data rows <tbody>. In practice, the table footer block <tfoot> block is not commonly used, and will not be discussed further. However, to read more about its usage, see http://www.w3schools.com/tags/tag_tfoot.asp. The "best practice" information at the end of this section describes why it can be beneficial to all users to add these blocks.

<table>
	<caption>Table description goes here</caption>
    <thead>...table header block...</thead>
	<tfoot>...optional footer block...</tfoot>	
	<tbody>...main table data block...</tbody>
</table>

A critical step toward creating a well formed, presentable and accessible data table is to designate column and/or row headers. In the markup, the <th> element is used for table header cells and the <td> element is used for table data cells. To separate each row in the table, the <tr> element is used. Using <th> elements to indicate the header cells will not only provide semantic meaning to the cell, but also apply a style for visual users.

<table>
<caption>Table of chemical elements</caption>
<thead>
    <tr>
        <th scope="col">Name</th>
        <th scope="col">Symbol</th>
        <th scope="col">Atomic number</th>
    </tr>
</thead>
<tbody>
    <tr>
        <th scope="row">Carbon</th>
        <td>C</td>
        <td>6</td>
    </tr>
    <tr>
        <th scope="row">Cesium</th>
        <td>Cs</td>
        <td>55</td>
    </tr>
</tbody>    
</table>

Note that whilst <th> elements are usually used to indicate the meaning of the data in the column, <th> elements can also be included in the <tbody> block where a header cell needs to indicate the meaning of the row. This mixture of headers used for both columns and rows is illustrated in the following table:

Table of chemical elements
Name Symbol Atomic number
Carbon C 6
Cesium Cs 55

However, to aid accessibility of the table, we should explicitly tell the browser whether the <th> element is a header for the column or the row and for this we add the scope attribute to each <th> element.

Use: <th scope="col"> to indicate a <th> element applies to the column.
Use: <th scope="row"> to indicate a <th> element applies to the row.

Best practice

  • Never use HTML table to control the layout of the page - there are other and better approaches. Only use tables to present tabular information in a grid or matrix.
  • Add a <caption> element to a table helps all users and gives a more professional appearance to your document. This must be the first thing after the opening <table> tag.
  • A critical step toward creating an accessible data table is to designate column and/or row headers using <th> elements. Doing so adds the all important semantic information and also applies a visual style which will help all users.
  • Including <thead> and <tbody> (and optionally <tfoot>) elements to indicate blocks within the table may benefit all users. Where a table is very long and stretches over more than one page if printed, some web browsers use this semantic information to ensure that rows in the <thead> block are repeated at the top of each printed page - and likewise <tfoot> blocks are repeated at the bottom of each printed page. Wrapping the <tbody> element around the main table data opens up the possibility of allowing the main table data to scroll independently within the table (though not by default within this template).
  • Avoid using the table elements <colgroup> and <col> as they often add complexity to table layouts which may not be supported by assistive technologies such as screen readers.
  • An excellent guide to explaining best practice for using and structuring HTML tables is available at http://webaim.org/techniques/tables/data.

Box placement

This template provides simple styles to aid positioning of boxes within a document. Controlling the position of elements with any more precision than is provided here soon becomes unwieldy and complicated. This template simplifies positioning down to either left, right or center. The US spelling of "center" is deliberate as the underlying language of styling (CSS) uses this spelling.

In the case of positioning left and right, the style actually positions the box left or right and forces any other content, e.g. text content, to "flow" around it. The underlying style which does this is called "float" which is why the styles illustrated below are called "float-left" and "float-right". This is illustrated better in the section about Figures where the float-left and float-right styles are used to position images with text flowing around them. However, the use of the float styling rule can introduce some problems with layout with boxes intruding into content below or pulling other content out of position. See the section on Layout for how to fix such issues.

When using the "center" style, text cannot be forced to flow around both sides and instead is interrupted by the center placed box element.

Box widths

The width of a box can be controlled using styles from one of two sets of width styling rules. Width rules specified with pixel units are fixed widths and keep their dimensions if the browser window is resized. Width rules specified with percentage units are widths relative to the width of the browser window and will change their width if the browser window is resized to ensure they only occupy a "percentage" of the available document width.

In both cases, rules exist to ensure boxes have a sensible minimum width and where the device being used to view the document has a small screen, box width is reduced to fit the available space.

Best practice

When using icon boxes, always ensure you have a clear warning text message rather than just relying on the icon image to convey the meaning of the message. Figures - Displaying image figures and figure captions

Figures, such as images within documents, should be enclosed within the <figure> HTML element. Where possible, always add the <figcaption> element inside the <figure> element and supply a description of the image. Note that it is also acceptable to add more than one image within a single <figure> element and then provide a <figcaption> to describe all the images. However, each individual image should have its own "alt" description - see the "Best practice" information below.

Within this template, all images with automatically scale according to the available screen space on the device they are viewed - this helps to make the content work better on smaller tablet and mobile screens. This is known as "responsive" behaviour i.e. they "respond" to the screen size they display on.

The modern HTML5 way to present document image figures is as follows:

<figure>
    <img src="500x200_sample_image.jpg" alt="A sample 500 x 200 image." />
    <figcaption>A sample 500 x 200 image.</figcaption>
</figure>

Here the figure contains multiple images. Each image has its own individual alt tag with one <figcaption> to describe the whole figure. 

<figure>
	<img src="resources/sample1.jpg" alt="Photo of  galloway cattle." />
    <img src="resources/sample2.jpg" alt="Photo of the common kingfisher." />
    <img src="resources/sample3.jpg" alt="Photo of the greylag goose." />
    <figcaption>Holiday photos from 2014.</figcaption> 
</figure>

You can also add similar styles to those for "boxes" to the <figure> element, such as styles to control appearance, width and position.

Best practice

Always try to add the <figcaption> element inside the <figure> element. This is not only the recommended HTML5 approach to web accessibility, but will also aid all users by providing extra meaning to the image. If is not appropriate to include a visual text figure caption, you should still add an "alternative text" to the image tag, e.g. <img src="myimage.jpg" alt="A sample 200 x 200 image." />.

If an alt description is largely redundant, for example when images are used purely for decorative purposes, still add the alt attribute, but leave "empty quotes" (null). This marks the image such that assistive technologies know to ignore it. e.g.

<img src="print-icon.jpg" alt="" />.

Note: Have a "null" alt attribute is not the same as having no alt attribute.

Adding alt descriptions and figcaptions is the first principle of web accessibility. An excellent guide to explaining how, why and best practice (with examples) is available at http://webaim.org/techniques/alttext/. Also see: https://dev.w3.org/html5/alt-techniques/.

Layout

This template has been specifically designed so keep the underlying HTML layout clean, useable and tailored towards linear a document style for teaching notes. The HTML template uses some HTML5 semantic tags to define different parts of the web page. These contribute to making the document more accessible and generally provide a clearer underlying structure. Whilst there is some discussion about semantic elements in this section, it is not essential to focus on these as there are great usability and accessibility gains to be had by following other best practice and would be doing nothing wrong by adding content directly to the <main> block of a template page.

Semantic elements

A semantic element clearly describes its meaning to both the browser and the web author.

  • Examples of non-semantic elements: <div> and <span>
     Tells nothing about its content.
  • Examples of semantic elements: <table> and <img>
    Clearly defines its content.

HTML5 offers new semantic elements to define different parts of a web page. This template is defined with a number of key semantic elements: <header>, <nav>, <main> and <footer>. Together, they create the structure in Figure 1. In addition, this guide has already covered the HTML5 semantic elements <figure> and <figcaption>.

If the text and image content for the document were added directly the <main> region, it would not be incorrect. However, there are further semantic elements to help give additional meaning to content.

Within a document used for teaching materials there may be natural sections. For precisely this, HTML5 includes the <section> element. Use of this element will not affect the styling in the default template styles, but additional styling can be added to distinguish sections if necessary.

Where a piece of content is not the primary focus of the document, but is still related to the document, the <aside> element may be used. The aside content should be related to the surrounding content, but not dependent on it so that it could be removed from the document entirely, placed on a page of its own, and it would still make sense. An <aside> element can be be placed within the <main> region or within a <section> region.

Note

Note that an <aside> semantic element can also be styled with the same styles as for boxes. For example, this callout box is using the <aside> element rather than the <div> element:
<aside class="box w350px callout2 icon-info">

Best practice

The introduction of HTML5 semantic elements is a step in the right direction to provide native "landmark" points within a document that assistive technologies can use to describe a document. However, both modern web browsers and assistive technologies are still to fully catch up with making full use of these new semantic elements.

WAI-ARIA provides the ability for web authors to specify roles for document areas (and many other things) is a way that screen readers understand. Such landmark roles have been included in the core areas of this document template for <header>, <nav>, <main> and <footer>. If you specify additional HTML5 semantic regions for <section> or <aside>, declare them with the following role information:

  • <section role="region"> ... <section>
  • <aside role="complementary"> ... </aside>

Further information about the relationship between HTML5 and ARIA landmarks is available at https://dequeuniversity.com/assets/html/jquery-summit/html5/slides/landmarks.html.

Fixing layout problems

Sometime positions of layouts don't do the correct thing and elements "spill" out of place. This is typically a problem when using the style for positioning boxes with "float-left" or "float-right" where the text flows around the box.

For example, the "right" floated box here has spilled down out of the current section. Add the following element with its "clearfix" style just before the content which you need to "clear" of the problematic box. For example, inserting the clearfix style at the end of this section (after the image figure) will force there to be clear space before the next part of the document continues and thus correct the layout problem.

<span class="clearfix"></span>

Mathematics - Display of Mathematical notation

This template utilises a well respected javascript library to help with the presentation of mathematical notation. This library is called MathJax and allows users to place mathematics directly into web documents using the mathematic subset of the LaTeX typesetting language or the web standard MathML. This provides many benefits over older approaches such as inserting images of mathematics into documents including providing high-quality typography both on screen and for print and accessibility advantages.

Using LaTeX notation

An equation displayed in a block can be delimited with double-dollar signs, e.g.

$$ \left( \sum_{k=1}^n a_k b_k \right)^{\!\!2} \leq \left( \sum_{k=1}^n a_k^2 \right) \left( \sum_{k=1}^n b_k^2 \right) $$

...will render as:

$$ \left( \sum_{k=1}^n a_k b_k \right)^{\!\!2} \leq \left( \sum_{k=1}^n a_k^2 \right) \left( \sum_{k=1}^n b_k^2 \right) $$

An equation displayed inline can be delimited with single-dollar signs, e.g.

This sentence contains inline mathematical notation such a $1 + \frac{q^2}{(1-q)}+\frac{q^6}{(1-q)(1-q^2)}$ which is part of the a Rogers-Ramanujan Identity equation.

...will render as:

This sentence contains inline mathematical notation such as $1 + \frac{q^2}{(1-q)}+\frac{q^6}{(1-q)(1-q^2)}$ which is part of the a Rogers-Ramanujan Identity equation.

Escaping dollar signs

If you have a need to display dollar signs without them being treated as MathJax delimiters, you can wrap the text with the tex2jax_ignore style to prevent MathJax from processing the content.

<p class="tex2jax_ignore">The equipment cost between $2 and $3.50 per item.</p>

...or you could even apply that style to a smaller portion using a <span> tag:

<p>The equipment cost between <span class="tex2jax_ignore">$2 and $3.50</span> per item.</p>

Using MathML notation

An equation that uses MathML as its source can be displayed in a block by adding display="block" to the opening <math> tag to give <math display="block">.

1 ( ϕ 5 − ϕ ) e 2 5 π = 1 + e − 2 π 1 + e − 4 π 1 + e − 6 π 1 + e − 8 π 1 + …

If the display="block" parameter is left out, the MathML display in-line as illustrated here:

This expression 3 x − 1 + ( 1 + x ) 2 is an example of an inline equation. As you see, MathJax equations can be used this way as well, without unduly disturbing the spacing between lines.

You can also explicitly specify the inline styling by adding <math display="inline">.

Best practice

The MathJax library relies on your knowledge of either LaTeX or MathML. If you need to rely on a visual (WYSIWYG) editor to author your mathematical notation, it is suggested you use one of the following options.

  • For LaTeX use the equation editor from CodeCogs:
    http://www.codecogs.com/latex/eqneditor.php
  • For MathML use the equation editor from Hermitech Laboratory called Formulator MathML Weaver:
    http://www.mmlsoft.com/index.php/products/mathml-weaver

Try to avoid including large quantities of mathematical notation in a single HTML document as this will slow down the display of the page as MathJax attempts to render all the mathematics it finds on the page.

Ensure that the mathematics notation you add as inline mathematics is as relatively small terms. Whilst "block" mathematics will line wrap to fit smaller device screens, inline mathematics will not and will cause problems with page layout on smaller screens.

If you have any specific MathJax requirements, you can tailor the configuration within the js/mathjax-config.js file within this template.

Media - Video and audio

Incorporating video clips

Whilst adding short video clips to your pages can really enhance the materials you are presenting, a video file is typically very large. It is recommended that you always utilise externally hosted streamed video. Most video produced within the University is hosted on https://stream.manchester.ac.uk and this service will provide you with the HTML markup to paste directly into your HTML page, though at time of writing, this HTML does not automatically scale down the embedded video area to fit smaller screens such as on mobile phones.

Where your video source is hosted outside of the University, you may be able to paste similar HTML markup from the source, e.g. YouTube or Vimeo. However, the HTML snippet that these site provide does not automatically scale down the embedded video area to fit smaller screens such as on mobile phones.

Embedding video from YouTube and Vimeo

This template includes a small number of styles to support embedded YouTube and Vimeo with automatic scaling to support different screen sizes. For both YouTube and Vimeo video, you can use the "embed" code snippet these sites provide with the following small changes.

  1. Paste the embed code into your HTML source and then remove the width and height attributes from the iframe tag.
  2. Surround the <iframe> embed snippet with two <div> elements as per the following example. Note the first of these <div> elements has the class style "embed-small"; the second <div> element has the class style "embed-container".
  3. Remember to close the <div> elements after the <iframe> element i.e. with closing </div> tags.

 

You can set the maximum size of the "embedded" video by changing the class style of the out enclosing <div> element between embed-small, embed-medium or embed-large. These will embed the video with the following maximum dimensions:

Embed style Maximum dimensions (width x height) in px
embed-small 225 x 400
embed-medium 360 x 640
embed-large 480 x 853

YouTube example:

<div class="embed-small">
	<div class="embed-container">
	<iframe src="https://www.youtube.com/embed/J5rwoBDHmOs?rel=0&amp;showinfo=0" frameborder="1" allowfullscreen=""></iframe>
	</div>
</div>

Vimeo example:

<div class="embed-medium">
	<div class="embed-container">
	<iframe src="https://player.vimeo.com/video/121325821" 	frameborder="0" webkitallowfullscreen mozallowfullscreen allowfullscreen></iframe>
	</div>
</div>

Embedding locally stored video files

Where you must link directly to a local video file, most browsers provide support for the MP4 video format. Use the <video> HTML5 markup to insert the video clip into your page:

<video width="360" height="240" controls>
   <source src="http://clips.vorwaerts-gmbh.de/big_buck_bunny.mp4">
Your browser does not support the video tag.
</video>

Incorporating audio clips

Audio clips are generally small enough to be practical to directly include within your web document resources folder. Not all audio formats are supported by all browsers. However, the MP3 audio format offers the widest support by far and is therefore the recommended format at present. To include an audio clip, use the following markup.

<audio controls>
  <source src="resources/horse.mp3" type="audio/mpeg">
  Your browser does not support the audio tag.
</audio>

 

Best practice

Within the <video> and <audio> tags, after the <source> declarations, include a short message for users of older browsers which don't support the particular media tag. For example, "Your browser does not support the video tag." or "Your browser does not support the audio tag."

  • Further information about use of the <video> tag is available at: http://www.w3schools.com/tags/tag_video.asp
  • Further information about use of the <audio> tag is available at: http://www.w3schools.com/tags/tag_audio.asp

Flash

The Adobe Flash technology has provided allowed for richer interative experiences on the web since the late 1990s. However, in recent years Flash has been plagued with security issues and usage has declined. Flash is not supported on mobile platforms and support on desktop web browsers is in decline. At the end of 2016, full support for Flash on the popular Chrome browser will be phased out. However, there remain many teaching resources developed in Flash which are still in use. For further advice, see the Best practice section for Flash.

Where Flash resources feature in the course materials, a special script should be used to "insert" the Flash resource into the page which will ensure that it runs smoothly for the end user.

Define a <div> element with an ID value to indicate the target location for the Flash resource. The <div> will be replaced with the correct markup to handle the display of the Flash resource. Beneath this <div> add the script code as indicated in the example below noting the meaning of each parameter.

Flash is not supported on your device. 
<div id="sample-flash1">Flash is not supported on your device.</div>
<script>
    swfobject.embedSWF("resources/tangramone.swf", "sample-flash1", 300, 200, 10);
</script>

Where:

resources/tangramone.swf =
relative path to the Flash resource;
sample-flash1 =
ID of the target <div> that the Flash resource will replace;
300 =
width of Flash resource;
200 =
height of Flash resource;
10 =
minimum target version of Flash that must be used to view the resource.

Best practice

Flash is a technology which is not native to the web browser and typically requires a browser plugin to be installed before it will display. It also has a growing reputation for introducing security problems, accessibility issues and is also demanding on platform resources. Flash is not supported on any modern mobile device platform (largely due to it being too demanding on available mobile computing resources). For these reasons, in recent years Flash has fallen out of favour as a technology with HTML5 technologies gradually replacing it.

Best practice advice is to avoid using Flash where possible and if it is utilised, provide an alternative resource which serves the same purpose.

Where Flash is used, within the <div> element where the Flash resource will be inserted, include a short message for users of browsers which don't support Flash. For example, "Flash is not supported on your device."
